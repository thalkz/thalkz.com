# The craft is just not required anymore
- Written in February 2026

> "We are part of the last generation of software developers. The job will survive —we still need people that are *responsible* for the tech, maintaining the product & making it evolve— but in a completely different shape. Especially, all the knowledge around the *craft* of writting code will disappear"

While sharing this take, I've have mixed feedbacks. Some say it is a bit too provocative: "the AI needs humains to know what to do, especially for large-scale things like architecture", and some say, on the opposite side, that it is a bit too conservative, that we won't even need humains overseeing product at all, or maybe even that the concept of a tech product will soon be outdated.

We discussed this recently with [Cyril](https://cyrilmottier.com/), at amo. He was questionning the bienfondé of the technical manifesto we were starting to write. This manifesto was supposed to assemble all coding principles that were applied at amo's Android team, some more largly applied at ythe company's scale. I do believe that this manifesto has solid ideas on what can make a small-to-medium tech team succeed. In my view, it is the state of the art of software engeneering.

*The state of the art of the old world.*

Is it still worth it to write it down ? That's not for certain. The *new world* of software engineering is entirely different from the *old world*.

In the *old world* writing code is costly. Devs have to write code manually, on a keyboard. Move fingers on top of keys, press them one by one. Move around files, and read things, painfully, slowly. Even with perfect vim gestures, this takes significant time. After each keystroke, look at the screen to see what the linter thinks, hover over that red squigly line, correct the typo. Try to run, see that it doesn't compile, read the error message, etc. 

And all of this doesn't even take into account the times it take to _know_ what to write. Learning the language(s), the framework(s), the codebase(s), the feature(s) to implement or the bug(s) to solve. Building this knowledge doesn't just take year, it takes whole career, entire lifetimes spent mastering this knowledge. Junior developers have a long road before them (well, _had_).

In my view, that road was enviable. One road paved with difficulties to overcome, but also eureka moments. It's fricking satisfying, to finally _get it_.

In this _old world_, reading and understanding code was a big part of the work, let's say 80%. The rest was about actually doing it, writing code. And because reading and understanding is so important, devs have created systems, patterns, good practices to keep code readable and easy to understand. It's all about getting the best possible ratio of writing / reading. Decades of knowledge have been accumulated to this end, in the form of books, articles, tools, frameworks, paradigms. What is means to be a "good developer" has been a key question for all.

So what happens when none of this is required anymore? Because that's today. Not "soon". Today.

Writing code now is basically free and fast. Reading code directly makes very little sense: an AI can do it better and will have less blindspots. Writing code that is optimised for humains doesn't make much sense anymore: we won't read it, an AI will.

I have no certainty on what this means for software development overall; but the craft of it will disappear.

It it just not required anymore.
